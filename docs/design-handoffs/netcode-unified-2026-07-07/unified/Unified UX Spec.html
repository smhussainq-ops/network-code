Netcode × Rez — Unified Network Operations UX

One console to discover, automate, verify, diagnose, and remediate network changes. Netcode is the shell and the only writer; Rez RCA lives inside it as a native, read-only Diagnostics workspace. Discovery, inventory and device state are shared; credentials stay runner-local.

Principles & boundaries

These are decided, not open. Every screen downstream obeys them, so review here first — a change to any of these ripples through the whole spec.

Netcode = shell + writer

Netcode owns the frame: identity, navigation, environments, and the only path that writes to devices. Every write goes through a gated workflow (plan → validate → apply → verify → rollback). There is no ad-hoc "run this command" button anywhere.

Rez = read-only diagnostics

Rez becomes a native Diagnostics workspace inside the shell. It observes, correlates, and recommends — RCA, forensics, digital twin. It never writes to a device. Its output can seed a Netcode change, but only as a proposal that re-enters the gated path.

Shared source of truth

Discovery, inventory and device/topology state are one dataset, produced once and read by both sides. No duplicate device lists, no divergent topologies. Diagnose and Automate reason over the same graph.

🔐
Credentials are runner-local — always.

Neither the shell nor Rez ever stores or sees device credentials. A local runner holds them and performs all device I/O (discovery reads, dry-runs, applies, rollbacks). The UI calls the runner and receives results and evidence, never secrets. This is the trust boundary that makes production adoption defensible.

🔁
The integration seam: Diagnose → Remediate.

The single most important new interaction. When Rez identifies a root cause, the engineer clicks "Create fix in Netcode". Rez emits a RemediationProposal; Netcode ingests it as a draft change that still must pass every gate. Read-only insight becomes gated action without Rez ever touching a device.

Operate

The write path. Discovery & Inventory (shared) → Workflow Packs (reusable automations) → Changes (the gated plan/validate/apply/verify/rollback pipeline). Everything here can lead to a device write, and every write is gated.

Diagnose

Read-only. Diagnostics/RCA (incidents, pcap forensics, the Rez copilot) and the Digital Twin (topology, route tracing, conflicts). Rez's old top-level modes — console · pcap · twin — collapse into these two sections. Nothing here writes.

Shared spine

One dataset. Discovery + Inventory + device/topology state sit under Operate for editing intent, but are read by Diagnose. Evidence & Audit unifies incidents (Rez) and changes (Netcode) into one timeline.

Why this structure
1
Grouped by intent, not by product. The engineer thinks "am I changing something or investigating something?" — not "am I in Netcode or Rez?" Operate vs Diagnose makes the read/write boundary the primary organizing line.
2
Mode is always visible. Operate items carry the amber write accent; Diagnose items the blue read accent. You always know if the current screen can touch a device.
3
Runner status is permanent chrome. Bottom of the rail shows runner connection + active environment. If the runner is down, every write action is visibly disabled — creds live there, not here.
4
Top bar (not shown): environment switcher (lab/prod), global search, and the primary New change action. Prod is gated behind lab proof.
5
Right context panel persists across screens (the Outcome panel): current object status, what's expected, next safe action, and the read/write boundary for what you're looking at.
02

Unified workflow

One operational loop spanning both products. Each stage names its owner and whether it can write. Diagnose is read-only and normally runs after Verify (post-change) or on demand (incident) — and it feeds Remediate, which re-enters the gated write path.

›
01
Discover
Shared · read
Scan subnet, auto-detect vendor, import devices + topology into the one source of truth. Runner does the reads.
›
02
Automate
Netcode · intent
Pick a Workflow Pack, define desired state, generate a Change Plan (diff + exact commands + risk). No device writes yet.
›
03
Verify
Netcode · gated
Policy + vendor syntax + lab dry-run must pass. Apply unlocks only on green. Post-apply, read back intended vs actual.
›
04
Diagnose
Rez · read-only
If something looks wrong (pre or post), open Diagnostics: RCA from telemetry/pcap, twin route trace, conflicts. Recommends only.
05
Remediate / Rollback
Netcode · gated
Rez's finding becomes a draft change, or roll back the last apply. Re-enters Verify. Loop closes with evidence.
↩︎
It's a loop, not a line.

Diagnose → Remediate hands a RemediationProposal back into Automate/Verify as a gated change — so a read-only diagnosis can never directly mutate a device, but it can propose a fix that follows the full safety path. Verify → Diagnose also runs automatically after every apply, correlating the change against live state to catch drift early.

Runner boundary

The runner is the only process with credentials. Both shells call it; it returns results + transcripts, never secrets.

🚦
Write authority is singular.

Only Netcode may call the runner with apply/rollback. Rez may call only read. The runner enforces this by op-type, so read-only is guaranteed server-side, not just in the UI.

05

Migration plan

Shell decision first, then move views by risk — shared spine before write path before diagnostics — so each phase ships something usable and reversible. Implemented surgically (per the checkpoint tag), slice by slice.

🏛️
Which shell hosts?

Netcode is the shell conceptually, but Rez's React/Vite stack (Cytoscape, rich state) is the stronger technical host. Recommendation: build the unified shell as a React app that consumes this design system. Netcode's workflow screens are simple and get re-implemented in React; Rez's Diagnostics/Twin move in largely intact, restyled read-only. The current static Netcode app.js becomes reference, not runtime.

0

Phase 0 · Foundation

Adopt the design system in both repos; freeze at checkpoint-2026-07-07-before-unified-ui-redesign. Stand up the React shell skeleton (nav rail, top bar, context panel) with tokens + components. No behavior moved yet.

1

Phase 1 · Shared spine

Move Discovery + Inventory first — lowest risk, read-mostly, and it's the single source of truth everything else needs. Collapse the two separate discovery UIs into one.

2

Phase 2 · Write path

Port the gated workflow: Workflow Packs → Change Plan → Verification → Apply → Evidence, preserving the runner boundary. Generalize add_vlan into the Pack model.

3

Phase 3 · Diagnostics

Embed Rez as read-only Diagnostics/RCA + Digital Twin, restyled. Collapse console·pcap·twin modes into the two nav sections. Gate chaos/lab tools behind the lab environment.

4

Phase 4 · The seam

Wire Diagnose → Remediate: RemediationProposal → pre-filled Pack → gated change. Auto-run a Diagnose correlation pass after each Apply. Unify Evidence across incident + change.

5

Phase 5 · Cleanup

Consolidate APIs behind one gateway; make read-only enforcement server-side in the runner. Retire Rez's localStorage-scoped incident tabs for the shared store. Delete dead code.

🧭
Sequencing rule.

Never move the write path before the shared spine, and never wire the seam before Diagnostics is read-only-clean. Each phase must leave the app shippable — if a phase can't ship, split it. Codex implements one slice per PR against the checkpoint tag.
